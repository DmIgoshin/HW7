# Стратегия развертывания ML-проекта

Этот репозиторий содержит пример ML-сервиса, развернутого с использованием стратегии канареечного развертывания (Canary Deployment).

## Стратегия канареечного развертывания

**Стратегия:** Канареечное развертывание было выбрано для минимизации рисков при внедрении новых ML-моделей, особенно учитывая чувствительность модели и отсутствие явной обработки ошибок. Эта стратегия позволяет тестировать новые версии (`app-green`) на небольшой части реального пользовательского трафика перед полным развертыванием.

**Начальное распределение трафика:**
*   `app-blue` (версия 1.0.0, стабильная): 90% трафика
*   `app-green` (версия 1.1.0, канарейка): 10% трафика

**Постепенное развертывание и откат:**
*   Трафик на `app-green` может быть постепенно увеличен путем изменения файла `nginx.conf` (например, с `weight=10` до `weight=30`, `weight=50` и, наконец, `weight=100`).
*   В случае проблем с новой версией, трафик может быть быстро перенаправлен обратно на `app-blue` путем корректировки `nginx.conf` до 100% для `app-blue` или удаления записи `app-green`.

## Структура проекта и файлы

*   `ml_pipeline.py`: Содержит основную логику обучения и предсказания ML-модели.
*   `app.py`: Простое Flask-приложение, которое обслуживает ML-модель. Включает проверки работоспособности и эндпоинты версий.
*   `Dockerfile`: Определяет, как собирается Docker-образ для Flask-приложения.
*   `nginx.conf`: Конфигурация Nginx для балансировки трафика между `app-blue` и `app-green`.
*   `docker-compose-canary.yaml`: Оркестрирует сервисы `app-blue`, `app-green` и `nginx` для канареечного развертывания.
*   `requirements.txt`: Список зависимостей Python.
*   `.github/workflows/ci.yaml`: Рабочий процесс GitHub Actions для непрерывной интеграции (тестирование `ml_pipeline.py`).
*   `.github/workflows/deploy.yml`: Рабочий процесс GitHub Actions для непрерывного развертывания (сборка Docker-образов и развертывание в облачном провайдере).

## Как запустить (Канареечное развертывание с Docker Compose)

### Предварительные требования
*   Docker и Docker Compose установлены в вашей системе.
*   Python 3.9+

### Шаги
1.  **Клонируйте репозиторий:**
    ```bash
    git clone https://github.com/DmIgoshin/HW7.git
    cd HW7
    ```

2.  **Убедитесь, что все необходимые файлы присутствуют:** Убедитесь, что `app.py`, `Dockerfile`, `nginx.conf`, `docker-compose-canary.yaml`, `ml_pipeline.py` и `requirements.txt` находятся в корневой директории.

3.  **Соберите и запустите сервисы:**
    Эта команда соберет Docker-образы для `app-blue` (v1.0.0) и `app-green` (v1.1.0) и запустит прокси-сервер Nginx, изначально маршрутизируя 90% трафика на `app-blue` и 10% на `app-green`.
    ```bash
    docker-compose -f docker-compose-canary.yaml up --build -d
    ```

4.  **Проверьте развертывание:**
    Вы можете получить доступ к сервису через Nginx на порту 80. Повторные вызовы эндпоинта покажут распределение трафика между версиями.
    ```bash
    curl http://localhost/
    curl http://localhost/version
    ```

5.  **Мониторинг и масштабирование трафика (необязательно):**
    Чтобы постепенно увеличить трафик на новую версию (`app-green`):
    *   Отредактируйте `nginx.conf` и настройте параметры `weight` в блоке `upstream backend`. Например, чтобы отправить 30% на green:
        ```nginx
        server app-blue:80 weight=70;
        server app-green:80 weight=30;
        ```
    *   Перезагрузите Nginx для применения изменений:
        ```bash
        docker-compose -f docker-compose-canary.yaml exec nginx nginx -s reload
        ```

6.  **Откат (при необходимости):**
    Если обнаружены проблемы с `app-green`:
    *   Отредактируйте `nginx.conf`, чтобы вернуть весь трафик на `app-blue`:
        ```nginx
        server app-blue:80 weight=100;
        server app-green:80 backup;
        ```
    *   Перезагрузите Nginx:
        ```bash
        docker-compose -f docker-compose-canary.yaml exec nginx nginx -s reload
        ```

## GitHub Actions CI/CD

### Непрерывная интеграция (`.github/workflows/ci.yaml`)
Этот рабочий процесс запускается при каждом push и pull request в ветку `main`. Он настраивает окружение Python, устанавливает зависимости и запускает `ml_pipeline.py`, чтобы убедиться, что основная логика ML работает.

### Непрерывное развертывание (`.github/workflows/deploy.yml`)
Этот рабочий процесс запускается при push в ветку `main`. Он собирает Docker-образ, отправляет его в GitHub Container Registry (GHCR), а затем запускает развертывание в вашем облачном провайдере с помощью команды `curl`.

**Важно:** Для корректной работы рабочего процесса `deploy.yml` необходимо настроить следующие [секреты GitHub](https://docs.github.com/en/actions/security-guides/encrypted-secrets):
1.  `CLOUD_TOKEN`: Токен аутентификации для API развертывания вашего облачного провайдера (это не токен GitHub).
2.  `MODEL_VERSION`: (Необязательно) Переменная для отслеживания версии модели, полезная для метаданных развертывания.
